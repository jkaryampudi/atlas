## The load-bearing finding, stated plainly (Stage 4b) — CONFIRMED against code

The deployed **live ranker** `atlas/dcp/signals/xsmom/generate.py` computes each name's 12-1 formation
return on **split-adjusted PRICE** closes — `adjust_for_splits(...)` then `c_skip / c_form − 1.0`
(`:218-222`), importing only `adjust_for_splits` (`:73`) with **no `total_return` import** — and writes
those winners under strategy family **`xsmom-pit-tr`** (`:82`). The approval that minted that same family
row ran in **TOTAL-return mode**: `atlas/tools/approve_xsmom_paper.py:105` calls
`run_xsmom_pit(total_return=True)`, and `atlas/dcp/backtest/xsmom_pit_run.py` only registers the
`xsmom-pit-tr` family when `total_return=True` (`:152, :643`), transforming the panel through
`total_return_series` — each dividend reinvested at its ex-date close (`:553-565`). **The live ranker and
the artifact it operationalizes therefore rank on different return conventions while sharing the name
`-tr` (total return).** This is **VERIFIED by exact-line inspection**, and the existing
`tests/unit/test_xsmom_generate.py` does not catch it — by inspection it pins the split-adjusted price
formula, so a green test *confirms* the price behaviour rather than flagging the divergence from the TR
approval. Whether the price↔TR gap materially changes the winner set is a data question this read-only
pass cannot answer (it needs the licensed dividend series) and is therefore **UNKNOWN** — but the
divergence itself is not in doubt.
